Como a internet funciona?

Ela começou a funcionar no Brasil comercialmente em 1995, mas já existia desde 1969, nos EUA. A partir disso, cada vez mais estamos migrando pra ela, estão se tornando raros serviços que não são prestados virtualmente. Com isso, a internet precisa ser rápida e nós somos as pessoas que irão "operá-la". Nesse site é bem legal poder ver a evolução das tecnologias através dos anos.A internet é uma mistura de cabos submarinos que atravessam o mundo todo. O tráfego de informações é intenso e constante.

Vou mostrar o que acontece com uma simples requisição pra um site.

O que significa Cliente e Servidor?

Um cliente pode ser o browser ou um telefone celular, por exemplo, ou até mesmo dispositivos IOT o wearables. Básicamente é alguma tecnologia que está entrando em contato com algum servidor na através da internet. Esse servidor, por sua vez, é só uma computador (muitas vezes bastante potente) que está recebendo essas requisições e processando conforme necessario.

O que é o Rails?

Um framework escrito na linguagem Ruby

O que significa que ele possui diversas gems (ou bibliotecas) que ele organizou de forma que seja simples para a pessoa simplesmente se preocupe com a implementação da lógica do negócio. Na própria documentação diz que o Rails "assume" o que é melhor pra você, porém, é totalmente opcional, podendo alterar como quiser. Existe até a doutrina do Rails, onde se fala sobre "convenção sobre configuração", e outros.

Estrutura básica de um projeto Rails

Usando o comando padrão de iniciar um projeto Rails, você tem a seguinte estrutura de pastas

A pasta APP

Iremos trabalhar nessa pasta a nossa maior parte do tempo. É nela que definimos os nossos modelos, as nossas regras de negócio, as nossas views (páginas com conteúdo que é exibida no browser do cliente).

A pasta config

Tirando o arquivo de rotas, dificilmente precisaremos alterar muitas coisas nessa pasta. É nela que adicionamos os nossos arquivos de internacionalização, onde configuramos o nosso projeto (a qual esperamos que seja totalmente exporádico).

Gemfile e outras pastas

O Gemfile é onde especificamos as dependências do nosso projeto (as gems que ele precisa pra funcionar). A pasta public é a pasta onde o Rails envia os arquivos preparados para o seu browser (ou do seu cliente) usarem, não modificamos o conteúdo dessa pasta a não ser que seja necessário.

Padrão REST e o MVC

REST é um padrão adotado para que as requisições sigam sempre o mesmo padrão de "verbo HTTP": GET (pra solicitar uma informação), POST (para enviar uma informação), UPDATE (atualizar), DELETE (apagar).

O Rails possui uma convenção de que, a partir do verbo que vêm na requisição, ele direciona para "ação" correta, daí o nome MVC > Model (Modelo), View (JSON ou uma pagina HTML), Controller (que possui Actions).

def show
    @aluno = Aluno.find(params[:id])
end

def create
    @aluno = Aluno.new params.require(:aluno)
    @aluno.save
end

Controllers

Como visto anteriormente, a Controller segue a convenção do padrão REST. E também por convenção, o Rails irá "procurar" um arquivo aluno.rb dentro da pasta models e uma pasta alunos dentro de views.
